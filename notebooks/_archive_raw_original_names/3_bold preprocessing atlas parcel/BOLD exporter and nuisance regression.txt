Plain-language documentation: BOLD exporter + nuisance model
What we’re trying to accomplish
We want a clean, consistent 200-parcel BOLD time series for each run, suitable for downstream HMM and EEG–BOLD fusion. The big challenges are:
	BOLD data has motion artifacts (sometimes subtle, sometimes catastrophic).
	We want to remove artifacts without throwing away lots of TRs, so we use regression-based “censoring” (spike regressors).
	We want stable parcel signals across reruns (same sign, same mapping, same outputs).
What files we use from fMRIPrep (per run)
	The BOLD time series in MNI space (*_space-MNI..._desc-*_bold.nii.gz)
	The matching MNI brain mask (*_space-MNI..._desc-brain_mask.nii.gz)
	Confounds table + metadata (*_desc-confounds_timeseries.tsv and .json)
Step-by-step: what the exporter does
Step 1 — Match BOLD and mask correctly
We only use a brain mask that matches the chosen BOLD file (same “stem” and space). This prevents silent mismatches that can ruin parcel extraction.
Step 2 — Put the atlas on the run’s BOLD grid
The Schaefer200 atlas is a labeled image (each voxel has a parcel ID 1–200).
But each run’s BOLD image can have slightly different voxel grid/affine. So we:
	resample atlas → BOLD grid using nearest-neighbor (so labels stay integers)
	save atlas_on_grid for transparency/debugging
Step 3 — Build the nuisance model (this is the “cleanup recipe”)
We build a matrix Xwith columns that represent stuff we don’t want in BOLD.
Then, for each voxel time series y(t), we fit:
y(t)≈X(t,:)β

and keep the residual as the cleaned signal.
What goes into X?
Think of Xas two groups:
A) Continuous confounds (smooth-ish nuisance signals)
These are time series columns that exist at every TR.
	Motion 24 parameters (motion24)
Why: head motion can affect BOLD in complex ways.
What: translations + rotations, plus derivatives and squares (a common “expanded motion” model).
	WM/CSF signals
Why: WM/CSF mostly reflect noise (physiology, scanner drift) rather than neural activity.
What: their signals + derivatives + squares.
	Cosine drift terms (cosine*)
Why: slow scanner drifts.
What: basis functions that capture low-frequency drift.
	aCompCor components (up to 10)
Why: data-driven noise components extracted from noisy tissue regions.
What: we prefer the components fMRIPrep marks as “Retained”.
	Non-steady-state outliers
Why: early TRs can be unstable (magnetization settling).
What: fMRIPrep provides one-hot columns for these.
All of these continuous columns are z-scored before regression (helps numerical stability).
B) Spike regressors (volume-specific “censoring without deleting TRs”)
Instead of deleting bad volumes, we add one-hot columns that absorb variance at specific TRs.
	FD spikes (FD > 0.5 mm)
Why: FD is a quick indicator of big head motion between volumes.
How: for TRs where FD > 0.5, add spike regressors (one-hot or small blocks).
	Catastrophic expansion (FD > 2.0 mm → also include t+1, t+2)
Why: a huge motion event can mess up not just that TR but the next couple TRs too.
	fMRIPrep motion_outlier* spikes ONLY when FD ≤ 0.5
This is the “FD-misses-it” safety net.
Why: sometimes there’s a big intensity transient that doesn’t show up as a big FD jump.
We saw exactly this in sub-03 at TR 19: FD was low, but nearly half the parcels went crazy. fMRIPrep flagged it as motion_outlier02, so we add that regressor only for low-FD outliers to avoid bloating the model unnecessarily.
Why we do NOT use DVARS spikes in the final model
DVARS can flag many TRs (sometimes 10–30%+) even in “good” runs, which can:
	add a huge number of regressors,
	remove real neural variance,
	reduce degrees-of-freedom.
So we keep DVARS-like measures for diagnostics, but we don’t automatically spike-regress them.
Step 4 — Parcel PC1 extraction (what “PC1 per parcel” means)
For each parcel, there are many voxels. We want one representative time series.
	If there are enough voxels, we do PCA and take PC1: the weighted combination of voxels that explains the most shared variance.
	If there are too few voxels, we take the mean (PCA isn’t stable with tiny voxel counts).
Why the “sign convention” matters
PCA has a sign ambiguity: PC1 and -PC1 are both valid.
So we force a stable rule:
	if PC1 is negatively correlated with the parcel mean, flip it.
This makes parcel time series consistent across reruns.
Step 5 — QC we run (and what each QC catches)
	FD summary → which runs have lots of motion or extreme spikes
	FD→PC correlation → whether motion is still leaking linearly into parcel signals
	Parcel blowup fraction → catches “many parcels spike at once,” including non-FD intensity transients
	Recompute-vs-saved QC → ensures the exporter is deterministic and reproducible (including sign)
